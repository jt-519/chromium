内核版本变更记录

M124 → M149 · 每版重大变更 · 官方来源

来源策略

本页记录的“内核版本变更”采用分层来源方案，不把单一页面误当成完整内核 changelog。

Chrome Release Notes 负责开发者可感知的 Web 平台变化；Chromium Dash 负责 milestone 时间线和版本映射；Chromium Source Log 负责源码提交区间审计；AndroidX WebKit 和 blink-dev 负责 Android WebView 特有差异与决策背景。

Chrome Release Notes
用途：提取 API、CSS、DOM、安全和媒体层的外显变化。
限制：不是 Chromium 源码级变更日志。
ChromeStatus
用途：补充 milestone feature 状态、规范和 issue 链接。
限制：不是提交级变更记录。
Chromium Dash
用途：确认分支点、稳定版时间和 milestone 对应关系。
限制：不描述详细行为变化。
Chromium Source Log
用途：深挖具体 commit 区间和源码变更。
限制：不适合直接面向业务做摘要阅读。
AndroidX WebKit
用途：记录 WebView 宿主 API、兼容开关和 Jetpack 配套变化。
限制：不能替代 Chromium milestone 主记录。
blink-dev
用途：补充 CHIPS、WebView 特例等决策背景。
限制：信息分散，适合作为补充来源。
版本变更记录工作流

下面的步骤按“版本映射 → 平台变化 → WebView 差异 → 源码核对”的顺序执行，适合人工维护，也适合 AI 按步骤汇总分析。

Step 1：建立版本映射

目标：确认本次升级跨越了哪些 milestone、stable 时间和版本区间。

操作：先查 Chromium Dash，记录 from milestone、to milestone、stable date 和版本映射关系。

入口：Chromium Dash

Step 2：提取平台外显变化

目标：整理开发者能直接感知到的 API、CSS、DOM、安全、媒体和运行时变化。

操作：遍历目标版本区间内各 milestone 的 Chrome Release Notes，只提取对升级评审有价值的 Added / Changed / Deprecated / Removed 项。

入口：Chrome Release Notes

Step 3：补充 Feature 状态

目标：为关键变化补充 feature 状态、规范、issue 和 WebView ship 信息。

操作：用 ChromeStatus 校验 milestone、补规范和问题单，避免只看 release notes。

入口：ChromeStatus

Step 4：补充 WebView 宿主变化

目标：把 Android WebView 的宿主 API、兼容开关和 Jetpack 配套变化单独整理出来。

操作：如果目标平台是 Android WebView，检查 AndroidX WebKit release notes，提取与 Cookie、拦截请求、导航、缓存、预渲染等相关的宿主变化。

入口：AndroidX WebKit

Step 5：补充 WebView 特例背景

目标：解释为什么某些行为在 WebView 与 Chrome 不完全一致。

操作：遇到 CHIPS、ship / unship、平台差异决策时，查 blink-dev，提炼决策背景而不是整段搬运讨论内容。

入口：blink-dev

Step 6：按需进行源码核对

目标：在需要追 commit、查根因或补证据链时，再做源码级核对。

操作：只有在 release notes 和 feature 信息不足以说明问题时，才下钻 Chromium Source Log，并将其作为证据补充而不是正文主体。

入口：Chromium Source Log

最终输出结构

完成以上步骤后，最终记录建议固定输出为以下结构，便于长期维护和横向对比：

  1. 版本映射
  2. 关键结论
  3. Breaking Changes
  4. Security / Privacy
  5. WebView Specific
  6. New Capabilities
  7. Runtime / Performance
  8. 来源说明
  9. 维护说明

输出规则：最终只保留对升级评审有价值的重大变化，不穷举所有提交，也不把单一 release notes 页面误当成完整内核 changelog。

维护说明

后续每次升核时，建议按固定顺序维护：先用 Chromium Dash 确认 milestone 和时间线，再用 Chrome Release Notes 和 ChromeStatus 拉平台变化主清单，然后用 AndroidX WebKit 与 blink-dev 补齐 WebView 特有差异，最后在必要时下钻 Chromium Source Log 做源码核对。

因此，本页中的每条记录默认代表“对升级评审有价值的重大变化”，而不是穷举所有提交。
