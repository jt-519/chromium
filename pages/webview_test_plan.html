推荐策略

  • 必测基准 — Speedometer 3.0、MotionMark、WebGL Aquarium
  • 业务页面性能 — Lighthouse CLI 本地采集 Core Web Vitals（详见 Page Perf 执行指南）
  • 兼容性与新 API — Browser Score、Can I Use
  • 媒体与网络 — WebRTC samples、WebCodecs demo
! 升核后最关键的是业务 H5 页面回归和 CPU / FPS / Memory 指标，第三方网站只能作为参考基准。
i 执行建议 — P1 与 P2 可并行；P3 依赖 P1/P2 通过；P4 依赖 P3 通过。全流程预估 4–6 周（含灰度）。
通用测试规范
  • 环境准备 — 关闭所有扩展/后台应用；飞行模式 + 屏幕常亮 + 亮度固定50%
  • 设备预热 — 开机 > 5 分钟，浏览器冷启动后空闲 30 秒再开测
  • 重复次数 — 每项基准跑 3 次，取中位数
  • 记录格式 — 设备型号 + 系统版本 + 浏览器/内核版本 + 日期 + 三次分数 + 中位数
  • 对比要求 — 同设备跨版本对比时，同一时段交替测试（排除温度差异）
  • 波动容差 — 同设备同环境连续跑测允许 ±3%；跨日期允许 ±5%

测试矩阵
  • 设备选取：低端机（4GB）+ 中端机（6–8GB）+ 高端机（12GB+） 必须
  • 系统版本：覆盖 Android 12 / 14 / 15，确保跨版本无回归
  • 每项基准运行 3 轮，取中位数；飞行模式 + 屏幕常亮 + 后台清空
必测项
  • Speedometer 3.0 — JS / 框架整体性能得分 必测
    测试操作点击 "Start Test" 全自动运行，禁止切换标签页 得分单位runs/min（每分钟完成的迭代数），越高越好 关注点总分 + 各框架子项分布，某框架子项异常低可能暴露 V8 优化缺陷 注意事项确保设备散热稳定；关闭所有扩展
    设备层级参考区间绝对红线
    高端移动(8Gen2+)10–18 runs/min≥ 8
    中端移动(6Gen1)6–12 runs/min≥ 5
    低端移动(4GB)3–7 runs/min≥ 3
  • JetStream 2 — V8 + WebAssembly 综合分 必测
    测试操作点击 "Start" 全自动，耗时约 3–5 分钟 得分单位综合几何平均分（geometric mean），越高越好 关注点总分 + Worst 5 子项 + Wasm 类子项单独对比 注意事项对 JIT Tier-up 敏感，冷启动 vs 热启动差异大；内存压力下 GC 子项劣化 特殊规则任意单子项劣化 > 30%，即使总分通过，仍需提交根因分析
    设备层级参考区间绝对红线
    高端移动120–200≥ 95
    中端移动70–130≥ 55
    低端移动40–80≥ 35
  • MotionMark — 渲染 / 动画帧率得分 必测
    测试操作选择 "Run Benchmark" 默认套件，约 2 分钟完成 得分单位score（60FPS 下能绘制复杂对象的加权几何均值），越高越好 关注点Multiply 子项(CSS合成层) + Canvas Lines(2D光栅化) + 帧率曲线平滑度 注意事项对 GPU 驱动版本极其敏感；分辨率/DPI 直接影响得分；统一窗口大小
    设备层级参考区间绝对红线
    高端移动(Adreno 740)400–900≥ 300
    中端移动(Adreno 619)150–400≥ 120
    低端移动(Mali-G52)80–200≥ 60
  • WebGL Aquarium — 3D 场景 GPU 渲染压力 FPS，默认 1000 鱼，建议梯度测试 1000 / 5000 鱼观察帧率衰减 必测
    测试操作配置1：默认 1000 鱼，等待 10 秒稳定后读数 30 秒取平均 FPS
    配置2：调至 5000 鱼（URL 参数 ?numFish=5000），同样 10s 稳定 + 30s 读数     得分单位FPS（页面左上角实时显示），越高越好 衰减率公式衰减率 = 1 - (5000鱼FPS / 1000鱼FPS)

P2

Web 标准兼容性验证

确保升核后 API 支持不退化，新特性可用
↓
测试项
  • Kernel Test Suite — 内置 API 兼容性 + 性能回归测试，支持 CDP 自动化采集 必测
    采集方式CDP（Chrome DevTools Protocol）自动化驱动，支持无头模式批量执行 执行频率每次内核版本升级 / 关键 patch 合入后必须执行 0 容忍不允许出现相对上一版本的新增 Fail（除非有裁剪决策单）
    维度通过警告阻断
    API 兼容性通过率≥ 99.5%98%–99.5%（逐条 review）< 98% 禁止发版
    性能回归（vs baseline P50）≤ 5% 劣化5%–15%（记 Ticket）> 15% bisect 定位
    新增 Fail 用例0—> 0（无豁免）
    CDP 参数--headless --disable-extensions --disable-background-timer-throttling --enable-precise-memory-info
    5 次取 P50 / 单用例超时 30s / 整套超时 30min     性能判定跑 5 次取 P50（排除首次冷启动）→ 对比 baseline P50 → 劣化 > 15% 标记 REGRESSION → bisect 定位 commit → 修复后重新验证至 ≤ 5%
  • Browser Score — 属性支持度评分，不应退化 必测
    测试内容HTML5 / CSS3 / ES6+ / Web APIs / 安全特性的支持度检测（feature detection） 核心原则不退化 — 新版本得分 ≥ 上一版本得分 竞品基准目标：不低于上游 Chromium 同版本得分；关注与 Chrome Latest 的差距 注意事项工具更新可能导致满分变化（假退化）→ 同时记录满分基数；部分特性需 HTTPS；确保用 feature detection 非 UA sniffing
    退化处理步骤说明
    1. 导出子项对比标记所有"上版 ✓ → 本版 ✗"的特性
    2. 逐项确认原因主动裁剪 → 豁免 / 引擎副作用 → 必修 / 工具误报 → 验证后豁免
    3. 判定结论非豁免退化项 > 0 → 阻断发版
业务 API 重点检查
  • 对照 Chromium Dash 确认 feature flags 和 breaking changes 必查
  • 重点关注：SameSite 策略、Mixed Content 拦截、CORS 行为变更
  • Can I Use 作为查询参考——确认目标版本对业务关键 API 的支持状态（非执行测试）

P4

稳定性与灰度验证

长时间运行 + 灰度放量，收集线上指标
↓
稳定性测试
  • Monkey：adb shell monkey -p <pkg> --throttle 300 -v 50000 必测
  • 长时间浏览（30min+），监控内存泄漏 必测
  • 多 WebView 实例并发（5+），无 OOM 推荐
  • 弱网测试：3G / 弱 4G / 高丢包 推荐
灰度策略
  • 第 1 周：内部员工全量 → 收集 Crash / ANR / 性能指标
  • 第 2 周：外部灰度 5% → 对比大盘 P50 / P90
  • 第 3 周：放量至 20% → 确认平稳后继续
  • 第 4 周：全量发布 → 持续监控 7 天
回滚标准（任一触发即停止灰度并回滚）
  • Crash 率 > 对照组 × 1.2（相对增幅 20%）且持续 1 小时以上
  • ANR 率 > 对照组 × 1.5（相对增幅 50%）
  • 页面加载 P90 > 对照组 × 1.3（相对增幅 30%）且无法定位原因
