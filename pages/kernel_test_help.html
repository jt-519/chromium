快速上手

完整测试流程
  1. 在旧版 WebView（升核前）中打开 kernel_test.html
  2. 点击 Run All Tests，等待进度条跑完（约 30–60 秒）
  3. 点击 导出 JSON 保存结果文件（建议命名含版本号，如 chromium_118.json）
  4. 升级到新版内核后，重新打开 kernel_test.html
  5. 点击 导入基准，选择步骤 3 中保存的 JSON 文件
  6. 点击 Run All Tests，运行新版测试
  7. 点击 查看报告，自动显示新旧对比——退化项红色标注，改进项绿色标注
分 Tab 运行

每个 Tab 左上角有独立的 Run [Tab] 按钮，可单独跑某个模块快速验证。

注意：单独运行 Tab 会清空其他 Tab 的历史结果，导出前请确认已完成全量测试。

升核对比方法

推荐：使用工具栏「导入基准」功能——导入旧版 JSON 后运行新版测试，报告中自动生成新旧对比表格，无需手动脚本。基准数据保存在 localStorage('kernel_test_baseline')，跨会话保留。

如需在 CI/CD 中自动对比或自定义逻辑，可使用脚本方式：

// 对比脚本示例（Node.js） const old = JSON.parse(fs.readFileSync('chromium_118.json')); const neu = JSON.parse(fs.readFileSync('chromium_128.json')); // 按评分公式计算 API 能力分数（JSON 中无 score 字段，需手动计算） const weights = { localStorage:5, sessionStorage:5, indexedDB:5, fetch:5, websocket:5, wasm:5, webgl:5, canvas2d:5, crypto:5, xhr:3, worker:3, webgl2:3, webrtc:3, intersectionObs:3, resizeObs:3, mutationObs:3, serviceWorker:1, mediaRecorder:1, sharedWorker:1, offscreen:1, webgpu:1, broadcastCh:1, sharedBuf:1, notification:1, geolocation:1, perfObserver:1 }; function calcScore(basic) { let earned = 0, possible = 0; Object.entries(weights).forEach(([id, w]) => { if (basic[id] === 'N/A' || basic[id] === undefined) return; possible += w; if (basic[id] === 'PASS') earned += w; }); return possible > 0 ? 20 + Math.round(earned / possible * 80) : 0; } console.log('旧版得分:', calcScore(old.tabs.basic)); console.log('新版得分:', calcScore(neu.tabs.basic)); // 找出新版新增支持的 API Object.keys(neu.tabs.basic).forEach(id => { if (neu.tabs.basic[id] === 'PASS' && old.tabs.basic[id] !== 'PASS') console.log('✓ 新增:', id); if (neu.tabs.basic[id] !== 'PASS' && old.tabs.basic[id] === 'PASS') console.log('✗ 退化:', id); });

API 能力评分方案

📐评分范围与公式

评分仅基于 Basic Tab 的 26 项 API 能力，性能指标不参与计算。N/A 项（环境限制，非真正失败）同时排除出分子和分母，不会压低实际得分。

Score = 20 + round( earned_points / effective_possible × 80 )
基础分 20 + API 能力得分（最高 80）= 满分 100 · N/A 项不计入 effective_possible

其中 total_points = 核心(9×5) + 重要(7×3) + 扩展(10×1) = 76 分（N/A 项排除后实际可得分上限，earned_points 为实际通过项目分值之和）。

N/A 诊断指南

N/A 表示 API 在当前环境不可用。"预期 N/A"不需处理；"非预期 N/A"需结合 chrome://flags、Manifest 配置或 ROM 厂商定制排查。

API典型 N/A 场景排查方向
SharedArrayBuffer ✅ 预期：未开启 COOP/COEP 跨域隔离头 检查响应头 Cross-Origin-Opener-Policy: same-origin + Cross-Origin-Embedder-Policy: require-corp
SharedWorker ✅ 预期：Android WebView 全版本不支持 仅桌面 Chromium 可用，WebView 中不应依赖
WebGPU ✅ 预期：Chromium < 113 或未启用 --enable-unsafe-webgpu WebView 113+ 默认仍未启用，需厂商定制
Notification ✅ 预期：WebView 默认禁用通知 需宿主 App 实现 onPermissionRequest 桥接
Geolocation ✅ 预期：WebView 默认未授权定位 需宿主 App 实现 onGeolocationPermissionsShowPrompt
ServiceWorker ✅ 预期：Android WebView 默认禁用 需宿主调用 WebSettings.setAllowFileAccess(true) + 自定义 ServiceWorkerClient
WebRTC ⚠️ 排查：部分厂商 ROM 阉割了 RTCPeerConnection 对照原厂 Chromium 同版本验证；若仅本机失败考虑厂商定制问题
OffscreenCanvas ⚠️ 排查：Chromium 69+ 应支持，N/A 说明被定制移除 检查 ROM 是否裁剪 Canvas Worker 相关代码
WebGL2 / WebGL ❌ 异常：核心图形 API 不应 N/A 检查 GPU 驱动黑名单 chrome://gpu，或硬件加速被禁用
IndexedDB / localStorage ❌ 异常：核心存储 API 不应 N/A 检查私密浏览模式、磁盘配额、或宿主 App 的 WebSettings.setDomStorageEnabled

✅ 预期 表示环境性质决定，无需处理；⚠️ 排查 需对比原厂 Chromium 同版本表现；❌ 异常 必须修复，否则核心业务受阻。

JSON 结果中的得分字段

运行完成后，window.__KERNEL_TEST_RESULT__ 包含完整结果，可通过 CDP 读取：

{ "timestamp": "2024-11-01T08:30:00.000Z", "ua": "Mozilla/5.0 ...", "browser": { "chromium": "128", "android": "14", ... }, "summary": { "total": 45, "pass": 38, "fail": 2, "unsupported": 5, "completed": 45 }, "tabs": { "basic": { "localStorage": "PASS", "webgpu": "N/A", ... }, "performance": { "jsLoop": "142 ms", ... }, ... } }

注：得分由前端实时计算，JSON 结果中不存储 score 字段，需在读取后按公式计算，或直接对比 tabs.basic 中各 id 的值。

CDP 自动化接入

🐍Python / pyppeteer 示例
import asyncio, json from pyppeteer import launch async def run_kernel_test(): browser = await launch(headless=True) page = await browser.newPage() cdp = await page.target.createCDPSession() await cdp.send('Performance.enable') await page.goto('http://localhost/kernel_test.html') await page.click('#btnRunAll') await page.waitForFunction( '() => window.__KERNEL_TEST_RESULT__ != null', timeout=120000 ) result = await page.evaluate('() => window.__KERNEL_TEST_RESULT__') metrics = await cdp.send('Performance.getMetrics') # 计算 API 能力分数 weights = { 'localStorage':5, 'sessionStorage':5, 'indexedDB':5, 'fetch':5, 'websocket':5, 'wasm':5, 'webgl':5, 'canvas2d':5, 'crypto':5, 'xhr':3, 'worker':3, 'webgl2':3, 'webrtc':3, 'intersectionObs':3, 'resizeObs':3, 'mutationObs':3, 'serviceWorker':1, 'mediaRecorder':1, 'sharedWorker':1, 'offscreen':1, 'webgpu':1, 'broadcastCh':1, 'sharedBuf':1, 'notification':1, 'geolocation':1, 'perfObserver':1, } basic = result['tabs']['basic'] earned = sum(w for k,w in weights.items() if basic.get(k) == 'PASS') possible = sum(w for k,w in weights.items() if basic.get(k) not in ('N/A', None)) score = 20 + round(earned / possible * 80) if possible > 0 else 0 print(f"API 能力得分: {score}/100") print(json.dumps(result['summary'], indent=2, ensure_ascii=False)) await browser.close() asyncio.run(run_kernel_test())

常见问题

❓SharedArrayBuffer / SharedWorker 显示 N/A，是 Bug 吗？

不是。部分 Android WebView 配置中这些 API 被系统策略禁用（如 COOP/COEP 未启用），N/A 表示该环境不支持，不会影响评分。

❓运行单个 Tab 后，其他 Tab 结果消失了？

这是已知行为：单 Tab 运行会重置全局计数器和结果对象。如需对比多模块数据，请始终使用 Run All Tests，最后导出完整 JSON。

❓WebRTC / ServiceWorker 显示 FAIL，如何排查？

WebRTC DataChannel 在部分受限 WebView（如 flag --disable-webrtc）中会失败；ServiceWorker 在非 HTTPS 或 file:// 协议下不可用。确认页面协议和 WebView flags 配置后重测。

❓Stress 测试出现 TIMEOUT，正常吗？

低端设备或受限内存配置下属于正常现象。可通过 __KERNEL_TEST_CONFIG__ 降低 stressDomCount、stressIframeCount 或增大 stressTimeout 来适配设备能力。

❓打分 80 分，能发版吗？

80 分属于「待改进」区间。建议检查具体失分项是否与业务功能强相关：若失分全部来自 EXTENDED 1pt 扩展项，实际影响有限，可评估通过；若有 CORE 5pt 或 IMPORTANT 3pt 失分，需要分析根因再做决策。

❓如何跨版本对比两次测试结果？

内置对比（推荐）：旧核运行后导出 JSON；升核后在新页面点击「导入基准」选择该 JSON 文件，再运行测试。打开报告即可看到每项测试的新旧对比——退化项红色标注，改进项绿色标注，性能指标显示百分比变化（> 10% 高亮）。基准存储在 localStorage('kernel_test_baseline')，跨会话保留，点击「清除基准」可重置。

脚本对比：用于 CI/CD 或自定义对比逻辑，使用「使用指南」中的 Node.js / Python 示例对比两份 JSON 的 tabs.basic 字段。JSON 中不包含 score 字段，需按公式计算。

此外，每次测试结果会自动写入 localStorage('kernel_test_result')，可通过开发者工具快速取回最近一次结果，但不保留历史记录（每次运行会覆盖）。
